# 项目知识问答与文档维护

> 基于项目知识库和源代码回答问题，并将新知识同步回知识库文档。

## Trigger

TRIGGER when: 用户询问项目相关的技术问题、模块功能、接口实现、代码链路、业务逻辑等
DO NOT TRIGGER when: 通用编程问题、非项目相关的问题

## Instructions

当用户提出项目相关问题时，按照以下三步流程执行：

### Step 1: 拉取最新代码

```bash
# 尝试拉取主仓库和子模块的最新代码
git pull || echo "网络问题，跳过拉取"
git submodule update --remote --merge 2>/dev/null || echo "子模块拉取失败，使用当前版本"
```

如果网络问题导致拉取失败，继续执行后续步骤，不要阻塞用户。

### Step 2: 基于知识库和源代码回答问题

**知识库位置**: `docs/knowledge/`

**源代码位置**: `sources/` (各子模块)

回答问题时：

1. **先查知识库**: 使用Glob搜索相关文档
   ```bash
   # 搜索相关专题文档
   Glob docs/knowledge/**/*.md
   ```

2. **再查源代码**: 在sources下搜索相关实现
   ```bash
   # 搜索源代码中的实现
   Grep pattern sources/{module}/
   ```

3. **综合回答**: 结合知识库文档和源代码实际实现，给出准确答案

### Step 3: 更新知识库并提交

回答完成后，如果发现了新的知识点或需要更新文档：

1. **确定更新内容**:
   - 新发现的代码实现细节
   - 接口调用链路
   - 模块边界澄清
   - FAQ和常见误解

2. **选择目标文档**:
   - 如果是专题内容，更新对应的专题文档
   - 如果是新专题，创建新文档并在MOC中添加入口
   - 遵循知识库的文档格式规范（Obsidian frontmatter）

3. **提交变更**:
   ```bash
   git add docs/knowledge/
   git commit -m "docs: 更新知识库 - {更新内容摘要}"
   git push || echo "推送失败，下次再推"
   ```

## 文档格式规范

知识库文档使用Obsidian格式：

```markdown
---
tags:
  - knowledge-base
  - {模块名}
status: active
updated: {YYYY-MM-DD}
---

# {文档标题}

## 问题背景
...

## 核心内容
...

## 依赖关系
...

## FAQ
...

## 待确认项
...
```

## 注意事项

1. 回答问题时，优先引用知识库中已有的结论
2. 如果知识库与源代码有矛盾，以源代码为准并更新知识库
3. 更新文档时保持专题文档的独立性，避免大杂烩
4. MOC文档只做索引，详细内容写在专题文档中
