# Mof-Web-Fe 接口知识库 MOC

## 1. 这组文档的用途

这组文档用于把 `mof-web-fe` 这个前端代码仓库整理成可长期维护的知识库，服务于后续系统重构、前后端边界重建和异语言重写。

这套知识库重点回答四类问题：

1. `mof-web-fe` 在整个系统里到底扮演什么角色。
2. 前端页面、报告、卡片、轮询、缓存是如何组织起来的。
3. 它依赖了哪些后端域和哪些稳定契约。
4. 未来如果重写前端或拆分 BFF，哪些行为不能丢。

## 4. 当前已确认的核心结论

- `mof-web-fe` 是整个系统的前端聚合门户，不是只服务单一业务域的 SPA。
- 仓库名虽然是前端仓，但 `package.json` 里的包名仍然叫 `mom`，说明其历史上与 `mom` 业务域深度绑定。
- 路由主干仍由 `js/pages/nav.ts` 控制，但同时并入了 `js/v2/routes.ts`，形成“旧页面树 + 新报告/组合页”的双代架构。
- 全局运行时仍以 `Redux + Saga + Connected Router` 为主，但新页面已经开始局部使用 `zustand + hooks`。
- 数据请求层已经形成三层：
  - `js/app/ajax` 的通用请求底座
  - `js/api/call-api.ts` 的统一结果解析封装
  - `js/api/*` 下按后端域拆分的 typed API 包
- 前端明确依赖多个后端域，不只依赖 `mof_api`：
  - `mof_api`
  - `rof_api`
  - `parser_api`
  - `indicator_api`
  - `research_api`
  - `monitor_api`
  - `rrp_api`
  - `rrp_app_api`
  - `info_api`
  - `rrp_market_adv`
- “报告/卡片平台”是这个仓库最值得保留和抽象的能力层，因为它串起了卡片配置、异步计算、轮询、缓存、导出预览和模板编排。
