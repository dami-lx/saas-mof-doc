# 模拟组合情景回测与报告专题

## 1. 这个专题关注什么

这篇文档只看模拟组合的结果计算与报告展示链路。

它覆盖：

- 模拟组合报告页
- 持仓绩效与回测触发
- 自定义情景回测归因
- 任务轮询与结果展示

## 2. 前端页面入口

关键路由：

- `/app/scenario/report/:id`
- `/scenario/report/:id?`

主要目录：

- `mof-web-fe/js/pages/scenario_new/report`
- `mof-web-fe/js/v2/pages/IndependentReport`

这里要注意两种展示形态：

- 老的 `scenario_new/report` 页面
- 新的 V2 报告页 / 卡片页

## 3. 前端 API 入口

关键文件：

- `mof-web-fe/js/pages/scenario_new/report/actions.ts`

已确认接口：

- `GET ${mof_api}/api/simulate/position/performance/{id}`
- `POST ${mof_api}/api/attribution/simulation/v3/custom`
- `GET ${mof_api}/api/task/{taskId}`

前端实现语义：

1. 先拿模拟组合的持仓绩效上下文
2. 根据返回里的 `needSimulation` 决定是否继续发起回测
3. 如果需要回测，则把：
   - `benchmark`
   - `startDate`
   - `endDate`
   - `navFrequency`
   - `holdings`
   - `curHolding`
   组装成 `simulation/v3/custom` 请求
4. 再通过 `task` 轮询拿最终回测结果

## 4. `mom` 后端入口

### 4.1 回测归因入口

主 controller：

- `mom/mom-legacy/src/main/java/com/datayes/web/mom/controller/AttributionController.java`

已确认接口：

- `POST /api/attribution/simulation/v3/custom`
- `POST /api/attribution/simulation/v3/custom/rx`

这说明模拟组合报告中的回测归因，核心仍在 legacy attribution 链上。

### 4.2 回测执行链

当前已确认下游实现锚点：

- `mom/mom-web/src/main/java/com/datayes/web/uranus/service/SimulationProcess.java`
- `mom/mom-web/src/main/java/com/datayes/web/mom/pk/simulate/SimulationComputer.java`

说明：

- `simulation/v3/custom` 不是一个薄 controller
- 它背后已有专门的 simulation process / computer

### 4.3 结果包装中的归因补全

当前已确认共同模型：

- `SimulationCustumPostV3`
- `NavAttributionManager`

这意味着模拟组合报告不仅要算净值序列，还会继续补充：

- 净值归因
- 相关 benchmark 的 attribution 结果

## 5. 数据来源与外部依赖

### 5.1 当前可确认依赖

- `AttributionController`
  - 自定义情景回测入口
- `SimulationProcess`
  - 回测任务编排
- `NavAttributionManager`
  - 净值归因填充

### 5.2 当前未完全确认但需要保留的问题

前端第一个请求：

- `GET /api/simulate/position/performance/{id}`

当前在仓库中还没有直接定位到 controller 入口，但前端明确依赖该接口返回：

- `holdings`
- `benchmark`
- `startDate`
- `endDate`
- `navFrequency`
- `latestPositions`
- `needSimulation`

这部分需要后续继续补 controller 或 service 锚点。

## 6. 公式与口径

这一专题是“真正的计算型链路”。

当前可确认的稳定口径：

- 报告页先拿持仓上下文，再决定是否跑回测
- 回测输入的持仓是时点序列，不是单期静态仓位
- `benchmark`、`navFrequency`、`curHolding` 都属于必需计算上下文
- 归因并非报告后处理装饰，而是结果的一部分

## 7. 重构时必须保留的契约

- 报告页的“预查询上下文接口”和“正式回测接口”要继续分开。
- `needSimulation` 这样的短路逻辑不能丢，否则会重复跑不必要计算。
- 结果里要同时保留净值表现和 attribution 结果。
- 不要把 legacy attribution 链直接忽略，它现在仍是核心实现。
