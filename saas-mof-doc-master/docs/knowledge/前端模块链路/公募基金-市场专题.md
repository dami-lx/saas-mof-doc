# 公募基金市场专题

## 1. 这篇文档解决什么问题

公募基金模块里，`/app/fund/market/public` 不是一个“列表 + 单接口”的普通页面，而是一个按专题卡片组织的数据看板。

这篇文档重点回答：

- 页面从哪里进入。
- 左侧导航如何组织卡片。
- 每张卡片主要调用什么 API。
- 这些 API 属于哪个服务域。
- 当前仓库里哪些实现能直接追到，哪些只是前端 wrapper。

## 2. 页面入口与整体结构

### 2.1 路由入口

关键路由：

- `/app/fund/market/public`

关键文件：

- `mof-web-fe/js/pages/fund-market/router.ts`
- `mof-web-fe/js/pages/fund-market/public-market/index.tsx`

页面结构很稳定：

1. `PublicDataWrapper`
2. `LeftNav`
3. `RightContent`

其中 `PublicDataWrapper` 在页面初始化时会统一拉取基金公司列表：

- `GET ${rof_api}/api/fund/market/company/all`

这说明市场页并不是每张卡片各自维护一份公共筛选元数据，而是页面级先取公共数据，再交给卡片消费。

### 2.2 左侧菜单不是路由，而是卡片目录

菜单定义在：

- `mof-web-fe/js/pages/fund-market/public-market/template.ts`

它把页面固定拆成四个一级分组：

- 持仓分析
- 指数基金
- 发行与上市
- 申购与销售

每个一级分组下面再挂 card key，最终通过 `all-cards.ts` 里的 `componentType / componentName / hash` 解析成左侧导航和页面锚点。

当前确认的卡片 inventory：

- `MarketPosition`
- `IndustryDistribution`
- `MarketSecurityDistribution`
- `HeavyWeightStocks`
- `HeavyWeightFunds`
- `InstitutionPositionTop`
- `IndexFundsValuation`
- `ETFSharesAndScale`
- `IssuingAndTrending`
- `FundEstablishment`
- `FundIssuing`
- `IssuanceDetails`
- `FundAudit`
- `PurchaseRedeem`
- `OrganizationSize`

这里有一个非常重要的重构结论：

- 公募市场页的稳定契约不是“页面 URL -> 一个 controller”
- 而是“菜单分组 -> card key -> card 自己的数据请求”

## 4. 后端实现边界

### 4.1 当前可确认的服务域

从前端常量可以明确确认：

- 公募市场页的 `apiHost = _config.rof_api`

也就是说，这一整块能力主域不是 `mof_api` / `mom`，而是 `rof_api`。

### 4.2 当前仓库没有直接看到对应 controller

截至当前工作区源码：

- 已明确定位到前端调用路径
- 已明确定位到大量 `/api/fund/market/*` 端点
- 但没有在 `mom`、`mom-robo`、`brain`、`pms` 当前目录里直接定位到这些 market controller

因此目前更稳妥的结论是：

- 公募市场页主要依赖外部 `rof` 市场数据服务
- 当前仓库更适合承担“入口梳理 + 请求协议梳理”
- 不应假设这部分逻辑实际在 `mom` 内部实现

## 5. 公式与数据口径

这块功能总体属于“专题统计与市场观测”，而不是 `brain` 那种算法计算链。

更值得沉淀的口径是：

- 报告期口径
- 公司维度与策略维度切换口径
- 发行统计与收益统计使用的时间粒度
- 卡片级筛选参数，如 `reportDate`、`statisticsFrequency`、`statisticsPoint`、`indexType`

换句话说，公募市场页的“重构难点”不是复杂数值公式，而是：

- 卡片拆分边界
- 多接口拼装关系
- 外部服务协议稳定性

## 6. 重构时必须保留的契约

- 保留“页面级公共数据 + 卡片级独立请求”的结构。
- 不要把公募市场页压缩成一个大接口，否则后续卡片增删会非常笨重。
- 保留 card key、componentType、hash 这套导航锚点体系，因为前端左侧导航直接依赖它。
- 对 `rof_api` 的 market 端点，要单独建立 adapter 层，不要和 `mom` / `brain` 的内部服务调用混写。

## 7. 后续补充建议

- 继续从 `FundAudit` 组件实现补齐实际请求。
- 对公募市场页所有 card 的查询参数做一张“卡片参数字典”。
- 如果未来拿到 `rof` 市场服务仓，可以再补 controller -> service -> 数据库 / 缓存 的后半段链路。
