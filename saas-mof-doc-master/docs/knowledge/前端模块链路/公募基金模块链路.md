## 1. 模块定位

这里的“公募基金模块”不是单一页面，而是一组围绕公募基金研究、筛选、市场观察和报告查看的能力集合。

当前可以分成三类入口：

- 基金筛选与列表：`/app/fund/product/filter/public`
- 公募市场观察：`/app/fund/market/public`
- 公募基金报告：`/report/:id(MUTUAL-.*)`、`/simple-report/:id(MUTUAL-.*|PRIVATE-.*)`

这三类入口共享同一业务对象，但底层链路并不相同：

- 列表页更偏搜索 / 筛选 / 排名
- 市场页更偏专题数据聚合
- 报告页更偏卡片装配和 `brain` 计算

## 2. 前端页面入口

### 2.1 路由入口

页面路由主要来自：

- `mof-web-fe/js/pages/nav.ts`
- `mof-web-fe/js/pages/product/router.ts`
- `mof-web-fe/js/pages/fund-market/router.ts`
- `mof-web-fe/js/v2/routes.ts`

当前确认的关键路由：

- `/app/fund/product/filter/public`
  - 页面目录：`mof-web-fe/js/pages/product/filter/publicV2`
- `/app/fund/market/public`
  - 页面目录：`mof-web-fe/js/pages/fund-market/public-market`
- `/report/:id(MUTUAL-.*)`
  - 页面容器：`mof-web-fe/js/v2/pages/IndependentReport/PageRouter`
- `/simple-report/:id(MUTUAL-.*|PRIVATE-.*)`
  - 页面容器：`mof-web-fe/js/v2/pages/simple-report`

### 2.2 用户可感知功能

从前端结构看，公募基金模块至少包含下面几类功能：

- 公募基金普通筛选
- 公募基金高级筛选
- 联想搜索 / 模糊搜索
- 筛选方案保存与复用
- 列表导出
- 公募市场专题卡片
- 公募基金评价报告 / 简版报告

## 2.3 当前已拆出的子域

为了避免把“筛选页”、“市场页”、“报告页”写成一锅，当前已经明确拆成三块：

- 筛选平台
- 市场专题平台
- 报告 / 卡片平台

其中市场专题能力见：

- [公募基金市场专题](./公募基金-市场专题.md)

## 3. 前端 API 入口

### 3.1 公募基金筛选页

核心 API 文件：

- `mof-web-fe/js/pages/product/filter/publicV2/service.tsx`
- `mof-web-fe/js/pages/product/filter/publicV2/components/AdvanceSearchPage/service.ts`

已确认的接口：

- `GET ${rof_api}/api/account/mutual/filter/field/dynamic`
  - 动态筛选字段
- `GET ${rof_api}/api/account/mutual/plans`
  - 读取筛选方案
- `POST ${rof_api}/api/account/mutual/plans`
  - 保存筛选方案
- `PUT ${rof_api}/api/account/mutual/plans/{id}`
  - 修改筛选方案
- `DELETE ${rof_api}/api/account/mutual/plans/{id}`
  - 删除筛选方案
- `POST ${rof_api}/api/account/mutual/associateSearch`
  - 联想搜索
- `POST ${rof_api}/api/account/mutual/advanceSearch`
  - 高级筛选

补充说明：

- 公募筛选页当前主要打到 `rof_api`，不是 `mof_api`。
- 这意味着“公募 / 私募外部基金筛选”在系统里与“投后 / 组合管理”是分域的。

### 3.2 公募基金报告页

报告页虽然从公募基金入口进入，但前端已经切到统一报告平台：

- 路由容器：`mof-web-fe/js/v2/pages/IndependentReport/PageRouter/index.tsx`
- 报告路由定义：`mof-web-fe/js/v2/routes.ts`

这类页面通常不会直接在页面文件里硬编码某个单一后端接口，而是进入统一卡片 / 报告运行时，再由卡片配置决定具体请求。

### 3.3 公募市场页

关键文件：

- `mof-web-fe/js/pages/fund-market/public-market/index.tsx`
- `mof-web-fe/js/pages/fund-market/public-market/template.ts`
- `mof-web-fe/js/pages/fund-market/public-market/cards/*`

当前结论：

- 页面整体数据域是 `rof_api`
- 页面先拉取公共基金公司列表
- 再由每张 card 各自请求专题接口

因此市场页不是单 controller 页面，而是一组 card 组合页。

## 4. `mom` 后端入口

### 4.1 公募基金筛选

主 controller：

- `mom/mom-web/src/main/java/com/datayes/web/mom/account/mutual/controller/MutualSearchController.java`

已确认接口：

- `POST /api/account/mutual/ordinarySearch`
- `POST /api/account/mutual/advanceSearch`
- `POST /api/account/mutual/dynamicSearch`
- `POST /api/account/mutual/app`

实现链条上的关键步骤：

1. `MutualReqPreProcessor` / `MutualAdvanceReqPreProcessor`
   - 默认分页与排序
   - 请求参数预处理
2. `MutualAccountRankServiceImpl` / `MutualAccountAdvanceRankServiceImpl`
   - 将筛选请求转换为带排序/排名语义的查询请求
3. `MutualAccountListService` / `MutualAccountAdvanceListServiceImpl`
   - 读取公募基金列表
4. `MutualResPostProcessor`
   - 结果后处理

这说明公募基金筛选不是简单 SQL 查询，而是：

- 前置参数清洗
- 排名/排序转换
- 列表查询
- 结果装饰

### 4.2 公募基金报告 / 卡片

主 controller：

- `mom/mom-web/src/main/java/com/datayes/web/mom/card/CardDataController.java`
- `mom/mom-web/src/main/java/com/datayes/web/mom/card/CardViewController.java`

其中 `CardDataController` 已明确提供：

- `/api/card/data/position`
  - 为算法调试构造持仓与基准入参
- `/api/card/data/nav`
  - 为算法调试构造净值与基准入参
- `/api/card/brainPost*`
  - 查看卡片最终发给算法层的请求

这对重构非常关键，因为它说明：

- 报告页真正稳定的契约，不只是“页面 -> 后端 JSON”
- 更重要的是“卡片模板 -> ReportParam -> Brain 入参构造”

## 5. 下游依赖与数据来源

### 5.1 列表筛选链路

从当前源码看，公募筛选链路主要依赖：

- `rof` 域的公募基金搜索与筛选服务
- `mom` 里的请求预处理、排名和结果后处理
- 账户/基金缓存与搜索能力

当前结论：

- 这条链路更偏“检索系统”，不是典型量化算法系统。
- 它的重构重点是字段语义、排序规则、筛选方案协议，而不是高性能数值计算。

### 5.2 市场专题链路

从前端实现已明确确认：

- 公募市场页主要依赖 `rof_api` 的 `/api/fund/market/*`
- 当前仓库未直接定位到对应后端 controller

因此这条链路的当前知识边界应定义为：

- 已定位页面结构、卡片结构、wrapper 和 API
- 未在本仓完成 controller / service / 数据库下钻

### 5.3 报告与分析链路

报告页下游会继续依赖：

- `mom` 卡片模板与报告框架
- `pms` / `pmsLite` 提供底层持仓、组合、净值等数据
- `brain` 提供绩效、归因、风险、风格等计算

推荐跳转：

- [Brain API 层与端点分类](../brain/Brain-API层与端点分类.md)
- [NavPerformanceOverviewTemplate 净值表现链路专题](../NavPerformanceOverviewTemplate-净值表现链路专题.md)

## 6. 公式与计算口径

### 6.1 列表筛选

公募筛选页本身以查询、筛选、排名为主，当前没有发现统一的“单个数值公式入口”。

更适合单独沉淀的口径包括：

- 排序字段来源
- 标签 / 主题 / 策略字段的取值口径
- 高级筛选字段的预处理规则

### 6.2 公募报告

公募报告中的公式并不固化在这个模块本身，而是分散在：

- `mom` 卡片模板
- `brain` 任务端点
- `mars` 等算法包

因此，当前知识库建议把“公募基金报告的公式”理解成对以下入口的索引：

- 卡片模板定义
- `CardDataController` 构造的算法入参
- `brain` 端点的计算口径

## 7. 重构时必须保留的契约

- 公募筛选页的筛选方案协议不能随意改，尤其是动态筛选字段、方案保存和联想搜索请求格式。
- 公募市场页的 card key、菜单结构和 `rof_api` adapter 边界要保留。
- 报告页不能只复刻页面展示，还必须保留“ReportParam -> Card -> Brain 入参”的中间层契约。
- `MUTUAL-*` 路由和普通页面筛选接口不是一条链路，重构时不要错误合并。

## 8. 待继续补充

- 公募市场页面的专题卡片与具体 API 对照表。
- `MutualAccountListService` 下游到底落到哪些缓存、ES 或数据库。
- 公募报告中常用卡片与 `brain` 端点的逐卡映射。

## 9. 当前拆解建议

公募基金模块后续推荐持续拆成两条主线：

1. 筛选平台
2. 报告平台

原因是这两条线虽然共享业务对象“公募基金”，但底层实现已经完全不同：

- 筛选更偏 `rof` 搜索与用户配置
- 报告更偏卡片平台和 `brain`
