# 业绩总览卡片项目实现与排查说明

## 1. 这篇文档解决什么问题

这篇文档是 [业绩总览卡片客户业务说明](./业绩总览卡片-客户业务说明.md) 的内部补充版。

客户版解决：

- 指标怎么给客户解释
- 图和表用了什么数据
- 展示前做了什么业务处理

本篇内部版解决：

- 前端从哪里进入这张卡片
- 前端交互如何触发 reload
- `mom` 和 `brain` 分别做了什么
- 字段到底代表什么
- 哪些地方最容易排查走偏

适用对象：

- 新接手投后 / AMS Pro 报告的人
- 排查业绩总览结果异常的人
- 准备做异语言重构或接口兼容的人

## 2. 功能档案

### 2.1 前端卡片身份

- 卡片名称：`业绩总览`
- card key：`performanceOverview`
- 前端 independent key：`navPerformanceOverview`

关键文件：

- `mof-web-fe/js/v2/cards/+productAnalysis/performanceOverview/config.card.ts`
- `mof-web-fe/js/v2/cards/+productAnalysis/performanceOverview/index.tsx`
- `mof-web-fe/js/v2/cards/+productAnalysis/performanceOverview/views/index.tsx`
- `mof-web-fe/js/v2/cards/+productAnalysis/performanceOverview/services.ts`
- `mof-web-fe/js/v2/cards/+productAnalysis/performanceOverview/quota.ts`
- `mof-web-fe/js/v2/cards/+productAnalysis/performanceOverview/definitions.ts`

### 2.2 前端可见交互

这张卡片至少有以下几类交互：

- Tab 切换
  - `收益走势`
  - `净值走势`
- 频率切换
  - `original`
  - `week`
  - `month`
- 时间区间切换
  - 预设区间
  - 自定义区间
- `过滤异常点`
- `包含转型前业绩`
- `年化因子设置`
- 指标选择

### 2.3 哪些交互真的会重新请求后端

前端会把以下参数打入 reload：

- `abnormalNavFilter`
- `customNavStartDate`
- `customNavEndDate`
- `transformationFundContinuation`
- `navSampleFrequency`
- `annualQ`
- `calendarType`

其中有一个容易误会的点：

- `过滤异常点` 只对 `收益走势` 曲线生效
- 代码里 tooltip 文案已经明确写死
- 所以看到图变而表不变，优先认为是设计口径，不要先怀疑底层公式

## 3. 前端对象模型

### 3.1 主 DTO

前端主接口数据结构定义在 `services.ts`，顶层对象可记成 `APIData`。

最重要的字段有：

- `portfolio`
  - 组合侧指标
- `benchmark`
  - 基准侧指标
- `navTimeSeries`
  - 净值走势图时间序列
- `nav`
  - 兼容老逻辑的收益或净值序列
- `benchmarkTimeSeries`
  - 基准收益序列
- `extReturnSeries`
  - 算术超额收益序列
- `activeReturnDivSeries`
  - 几何超额收益序列
- `originalNavFrequency`
  - 原始净值频率
- `monthlyNav` / `weeklyNav`
  - 周频/月频切换使用的时间序列
- `monthlyNavTimeSeries` / `weeklyNavTimeSeries`
  - 周频/月频净值图时间序列
- `transformDates`
  - 转型相关日期点
- `accumulateAverageAssetValue`
  - 累计平均持仓市值
- `accumulateAverageConcentration`
  - 累计平均持仓集中度
- `accumulateTurnoverRateYearly`
  - 累计平均持仓年化换手率

### 3.2 指标层字段

`portfolio` / `benchmark` 内部最常用字段包括：

- `annualTotalReturn`
- `annualActiveReturn`
- `activeAnnualReturnDiv`
- `annualTotalRisk`
- `annualActiveRisk`
- `totalReturn`
- `totalRisk`
- `calmarRatio`
- `sortinoRatio`
- `treynorRatio`
- `hurst`
- `stutzer`
- `realizedAlpha`
- `ir`
- `sharpeRatio`
- `winRatio`
- `profitLossRatio`
- `spearman`
- `pearson`
- `maxDrawdownRecoverDays`

### 3.3 基准和组合不是完全对称结构

`benchmark` 从字段上看和 `portfolio` 很像，但实现口径并不完全对称。

当前已确认：

- 基准端 `winRatio = 0D`
- 基准端 `profitLossRatio = null`
- 基准端 `spearman = 1D`
- 基准端 `pearson = 1D`

所以如果重构时把两侧当成完全同构对象处理，很容易引入兼容性偏差。

## 4. 整体调用链

## 4.1 前端层

前端卡片入口：

- `mof-web-fe/js/v2/cards/+productAnalysis/performanceOverview/index.tsx`

展示与交互主文件：

- `mof-web-fe/js/v2/cards/+productAnalysis/performanceOverview/views/index.tsx`

这层主要负责：

- 持有当前 tab、频率、时间区间、本地开关状态
- 组装 reload 参数
- 按当前产品类型决定是否显示几何超额收益
- 按 `quota.ts` 和 `definitions.ts` 做指标展示和说明

## 4.2 `mom` 模板层

核心模板：

- `mom/mom-web/src/main/java/com/datayes/web/mom/service/card/cardTemplate/common/NavPerformanceOverviewTemplate.java`

对应视图：

- `@CardView("navPerformanceOverview")`

这层主要负责：

- 读取 `ReportParam`
- 解析自定义参数，如 `abnormalNavFilter` 和 `annualQ`
- 调 `cardHelper.generateParam(reportParam)` 生成 `MultiIntervalNavPerfParam`
- 通过 `JBrainNavPerfComputerAdapter.generateComputer(...)` 构造净值表现计算器
- 计算组合端与基准端指标
- 打包 `NavPerformanceOverviewDto`
- 拼装前端需要的图表序列和附加字段

## 4.3 `brain` 接口层

已确认对应的下游端点包括：

- `/api/common/nav_performance_overview`
- `/api/nav_perf`

在 `mom` 中可见的调用声明位于：

- `mom/mom-web/src/main/java/com/datayes/web/mom/service/card/attr/CommonAttributionRptService.java`

在 `brain` 中可见的注册位于：

- `brain/lib/web_service/server.py`
- `brain/lib/web_service/main_service.py`
- `brain/lib/parallel_compute/tasks/task_mom.py`

### 4.4 底层计算框架

`NavPerformanceOverviewTemplate` 并没有手写所有收益风险公式，而是依赖：

- `JBrainNavPerfComputer`
- `JBrainNavPerfComputerAdapter`
- `NavPerfAdapter.getLoader()`
- `PerformanceOverviewCalculator`

因此这条链更接近：

- 标准化输入准备
- 通用净值表现计算
- 业务口径修正
- 结果 DTO 打包

而不是某个单独的“业绩总览算法函数”。

## 5. `mom` 这层实际做了什么

### 5.1 `compute(ReportParam)` 主流程

核心步骤可以概括为：

1. 读取 `abnormalNavFilter`
2. 读取 `annualQ`
3. 生成 `MultiIntervalNavPerfParam`
4. 构建 `JBrainNavPerfComputer`
5. 计算组合侧 `portfolio`
6. 计算基准侧 `benchmark`
7. 如果净值点不足，则返回 remark

### 5.2 组合侧 `computeWithPortfolio(...)`

当前已确认的关键行为：

- 先用 `computer.computeActiveAccumulateReturnSeries(false)` 计算累计主动收益序列
- 取末点写入 `accumulatedActiveReturn`
- 通过 `JBrainNavPerfComputerAdapter.computeNew(...)` 一次性计算指标集合
- 再用 `PerformanceOverviewCalculator.handleCalcParam(...)` 对自定义年化因子相关指标做覆盖修正
- 再设置：
  - `accumulateReturnSeries`
  - `spearman`
  - `pearson`
  - `winRatio`
  - `profitLossRatio`

这意味着：

- `annualQ` 不是改变原始净值
- 它更像在指标层做“展示口径修正”

### 5.3 基准侧 `computeWithBenchmark(...)`

关键行为：

- 先算基准表现指标
- 把 `param` 拷贝后替换成基准净值序列
- 再套一遍年化因子修正逻辑
- 设置基准累计收益序列
- 强制写死部分兼容字段

兼容字段固定值：

- `winRatio = 0D`
- `profitLossRatio = null`
- `spearman = 1D`
- `pearson = 1D`

### 5.4 视图层 `navPerformanceOverview(...)`

最终返回给前端的 `NavPerformanceOverviewDto` 会继续填充：

- `portfolio`
- `benchmark`
- `activeReturnDivSeries`
- `extReturnSeries`
- `navTimeSeries`
- `nav`
- `monthlyNav` / `weeklyNav`
- `monthlyNavTimeSeries` / `weeklyNavTimeSeries`
- `transformDates`

这说明 `业绩总览` 并不是只靠一个纯指标接口支撑，而是：

- 指标
- 收益序列
- 净值序列
- 转型日期

一起打包后的复合卡片。

## 6. 数据来源与口径

### 6.1 必需数据

主链计算至少依赖：

- 组合净值序列
- 基准净值序列
- 无风险利率
- 起止日期
- 净值频率

### 6.2 可选补充数据

如果前端勾选了某些指标，还会触发带 `holding_attr` 的 reload，补充：

- 平均规模
- 持券集中度
- 年化转手率

因此排查时要先分清：

- 是主业绩表现异常
- 还是持仓补充指标异常

### 6.3 异常点过滤口径

当前已确认：

- `abnormalNavFilter` 只用于收益走势曲线平滑
- 不会改掉其他指标和结果

因此如果用户说“只有曲线不一样”，先检查是不是这个开关造成的。

### 6.4 年化因子口径

`annualQ` 与 `calendarType` 会影响：

- 年化收益
- 年化风险
- IR
- Sharpe

更准确地说，它们影响的是：

- 部分指标的换算口径

而不是：

- 原始净值时间序列本身

### 6.5 转型前业绩口径

`transformationFundContinuation` 会影响：

- 可参与计算的净值历史区间
- 收益 / 风险指标
- 曲线形态

所以排查带“产品转型”的 case 时，不先确认这个开关，几乎一定会走偏。

## 7. 排查时最容易误解的点

### 7.1 误把“收益走势变化”当成“全卡片结果变化”

实际上：

- `过滤异常点` 只影响收益走势图
- 不是全卡片重算口径切换

### 7.2 误把 `annualQ` 当成底层净值清洗逻辑

实际上：

- 它是指标换算层修正
- 不是改了底层时间序列

### 7.3 误把基准字段当成组合字段的镜像

实际上：

- `benchmark` 有多处兼容常量值
- 不能简单按组合同一套逻辑重算后“自然得到”

### 7.4 误把附加持仓指标当成主链失败

实际上：

- 平均规模、持券集中度、年化转手率是补充请求触发的
- 它们为空不等于主收益风险链路失败

### 7.5 误把算术超额和几何超额混为一谈

前端明确区分：

- `extReturnSeries`
  - 算术超额收益
- `activeReturnDivSeries`
  - 几何超额收益

如果客户或测试拿两种口径直接逐点比对，通常会误判。

## 8. 推荐排查顺序

建议按下面顺序排查：

1. 先确认页面状态
   - 当前 tab
   - 当前频率
   - 当前日期区间
   - 是否勾选异常点过滤
   - 是否包含转型前业绩
   - 是否设置年化因子
2. 再确认异常类型
   - 只有图异常
   - 只有指标异常
   - 图和指标都异常
3. 再确认异常位于哪一层
   - 组合侧
   - 基准侧
   - 相对基准侧
4. 最后再下钻到模板或 brain

## 9. 重构时建议保留的契约

- 保留 `组合 / 基准 / 超额序列` 三套对象，不要只留单套指标
- 保留 `算术超额` 和 `几何超额` 的区分
- 保留 `benchmark` 兼容常量字段行为
- 保留 `annualQ` 作为单独一层口径修正，而不要混进原始净值层
- 保留 `异常点过滤只影响收益走势` 的现有契约
- 保留 `transformDates` 与频率切换支持，否则前端体验会退化
