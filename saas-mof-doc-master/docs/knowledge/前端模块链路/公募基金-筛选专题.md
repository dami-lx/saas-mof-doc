# 公募基金筛选专题

## 1. 这个专题关注什么

这篇文档只看公募基金的筛选链路，不讨论报告页。

它重点回答：

- 公募筛选页有哪些前端入口
- 普通筛选、高级筛选、动态字段、联想搜索、筛选方案分别走哪些接口
- 这些接口在 `mom` 里对应什么 controller
- 字段定义和动态搜索口径主要放在哪一层

## 2. 前端页面入口

关键路由：

- `/app/fund/product/filter/public`

主要目录：

- `mof-web-fe/js/pages/product/filter/publicV2`

从前端结构看，公募筛选页至少分成几块：

- 列表页
- 高级筛选弹层 / 页面
- 联想搜索
- 自定义列
- 筛选方案保存

## 3. 前端 API 入口

### 3.1 普通筛选与列表能力

当前页面代码里，普通筛选相关逻辑散在列表页状态和高级筛选里，但接口域已经比较清晰：

- `POST ${rof_api}/api/account/mutual/ordinarySearch`
- `POST ${rof_api}/api/account/mutual/advanceSearch`

### 3.2 动态筛选字段

关键文件：

- `mof-web-fe/js/pages/product/filter/publicV2/service.tsx`

接口：

- `GET ${rof_api}/api/account/mutual/filter/field/dynamic`

### 3.3 联想搜索

关键文件：

- `mof-web-fe/js/pages/product/filter/publicV2/service.tsx`

接口：

- `POST ${rof_api}/api/account/mutual/associateSearch`

### 3.4 高级筛选动态搜索

关键文件：

- `mof-web-fe/js/pages/product/filter/publicV2/components/AdvanceSearchPage/components/string-auto-complete/service.ts`

接口：

- `POST ${rof_api}/api/account/mutual/dynamicSearch`

这个接口会根据字段的 `dynamicSearchKey` 或 `dataIndex` 发请求。

### 3.5 筛选方案

关键文件：

- `mof-web-fe/js/pages/product/filter/publicV2/service.tsx`

接口：

- `GET ${rof_api}/api/account/mutual/plans`
- `POST ${rof_api}/api/account/mutual/plans`
- `PUT ${rof_api}/api/account/mutual/plans/{id}`
- `DELETE ${rof_api}/api/account/mutual/plans/{id}`

## 4. 字段定义与前端口径

公募筛选的字段定义并不只在后端，前端本身也沉淀了大量字段配置。

关键位置：

- `mof-web-fe/js/pages/product/filter/publicV2/self-columns/*`
- `mof-web-fe/js/pages/product/filter/publicV2/self-columns/index.ts`

当前可确认：

- 每个字段除了表格列属性，还可能带有：
  - `searchKey`
  - `dynamicSearchKey`
  - 自定义筛选组件
- 高级筛选动态搜索优先使用 `dynamicSearchKey`

这意味着未来重构筛选系统时，不能只复制后端接口，还要同步迁移前端字段元数据模型。

## 5. `mom` 后端入口

### 5.1 普通筛选 / 高级筛选 / 动态搜索

主 controller：

- `mom/mom-web/src/main/java/com/datayes/web/mom/account/mutual/controller/MutualSearchController.java`

接口：

- `POST /api/account/mutual/ordinarySearch`
- `POST /api/account/mutual/advanceSearch`
- `POST /api/account/mutual/dynamicSearch`

实现步骤：

1. `MutualReqPreProcessor` / `MutualAdvanceReqPreProcessor`
2. `MutualAccountRankServiceImpl` / `MutualAccountAdvanceRankServiceImpl`
3. `MutualAccountListService` / `MutualAccountAdvanceListServiceImpl`
4. `MutualResPostProcessor`

### 5.2 动态筛选字段

主 controller：

- `mom/mom-web/src/main/java/com/datayes/web/mom/account/mutual/controller/DynamicFilterFieldController.java`

接口：

- `GET /api/account/mutual/filter/field/dynamic`

已确认动态字段来源服务：

- `FundThemeService`
- `TracingIndexService`
- `IndustryCharacteristicsService`
- `FofBenchmarksFilterFieldService`
- `SecondaryStrategyFilterFieldService`
- `FundAwardService`

关键结论：

- 动态筛选字段不是临时查库结果，而是聚合多个 service 后返回。
- 注释明确写了这部分会进缓存，并通过 `DynamicFilterFieldCache` 刷新。

### 5.3 联想搜索

主 controller：

- `mom/mom-web/src/main/java/com/datayes/web/mom/account/mutual/controller/MutualFundAssociateSearchController.java`

接口：

- `POST /api/account/mutual/associateSearch`

### 5.4 筛选方案

主 controller：

- `mom/mom-web/src/main/java/com/datayes/web/mom/account/mutual/controller/QueryPlanController.java`

接口：

- `GET /api/account/mutual/plans`
- `POST /api/account/mutual/plans`
- `PUT /api/account/mutual/plans/{planId}`
- `DELETE /api/account/mutual/plans/{planId}`

实现语义：

- 筛选方案最终转成 `UserConfigDTO`
- 通过 `MutualQueryPlanService` 保存
- 本质上属于“用户配置模板”体系

## 6. 数据来源与外部依赖

### 6.1 当前可确认的数据层结构

- 前端访问的是 `rof_api`
- 当前仓库中的 `mom` 提供了同路径 controller 实现
- 动态字段服务已明确使用缓存
- 筛选方案归到用户配置存储模型

### 6.2 当前还未完全确认的部分

- 普通 / 高级筛选列表最终是走 ES、缓存还是数据库主查
- 排名服务里具体的排序字段翻译规则

## 7. 公式与口径

这一链路不是传统“公式计算”链，而是“筛选口径”链。

最重要的稳定口径是：

- 字段名与 `dynamicSearchKey`
- 动态字段枚举来源
- 排名与排序转换
- 用户筛选方案存储模型

## 8. 重构时必须保留的契约

- 动态筛选字段、联想搜索、方案 CRUD 不应被粗暴合并成一个 search API。
- 前端字段模型和后端字段语义必须成对迁移。
- 缓存刷新机制不能因为做接口重写而被忽略，否则筛选项会直接不稳定。
