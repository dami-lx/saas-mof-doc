# 资产配置落地组合与编辑专题

## 1. 这个专题关注什么

这篇文档只看资产配置方案“落地成组合 / 产品”之后的链路。

它重点回答：

- 研究结果如何真正创建成可管理组合
- 编辑、查询、删除分别落到哪些接口
- `assetallocation` 和 `netValue` 两条落地链分别负责什么
- 哪些接口已经进入 `PmsDataAccountMapping` 的双标识体系

## 2. 前端页面入口

关键路由：

- `/app/asset/edit/:id`

## 3. 前端 API 入口

### 3.1 资产配置组合落地

已确认接口：

- `POST ${mof_api}/api/assetallocation/creation`
- `GET ${mof_api}/api/assetallocation/editQuery?accountId=...&portfolioId=...`
- `POST ${mof_api}/api/assetallocation/edit/{reCal}`
- `DELETE ${mof_api}/api/assetallocation/delete`
- `GET ${mof_api}/api/assetallocation/modelDetails?accountId=...`
- `POST ${mof_api}/api/assetallocation/fund/list`
- `GET ${mof_api}/api/assetallocation/asset/scope`
- `POST ${mof_api}/api/assetallocation/mofData`

### 3.2 净值型组合落地

V2 组合页额外确认到：

- `POST ${mof_api}/api/netValue/creation`
- `GET ${mof_api}/api/netValue/editQuery?portfolioId=...&accountId=...`
- `POST ${mof_api}/api/netValue/edit/{reCal}`
- `DELETE ${mof_api}/api/netValue/delete`

关键结论：

- 资产配置方案落地后，并不只有一套编辑接口
- 至少区分：
  - `/api/assetallocation/*`
  - `/api/netValue/*`

## 5. 实现语义

### 5.1 创建阶段

`creation` 的关键实现特征：

- 需要 `momId`，缺失会直接报错
- 通过异步轮询返回 `PmsDataAccountMapping`

这里的 `PmsDataAccountMapping` 非常关键，它说明创建结果不是一个单纯的前端对象，而是：

- `mom account`
- `pms portfolio`

两套标识被正式绑定后的结果

### 5.2 编辑阶段

`editQuery` 和 `edit/{reCal}` 说明编辑不是无状态 patch：

- 页面打开时要先回查后端拼好的编辑视图对象
- 提交时要明确传 `reCal`

这意味着：

- 是否重算是稳定业务语义
- 不能在重构时简化成“字段变了就全量重算”

### 5.3 删除阶段

`/delete` 最终会委托：

- `ProductionAccountService.delete(accountId)`

这说明资产配置组合最终已经被纳入投后产品体系，而不是孤立的研究对象。

## 6. 数据来源与外部依赖

### 6.1 当前可确认依赖

- `AssetAllocationAccountService`
  - 自上而下资产配置组合创建、编辑、详情
- `NetValueAccountService`
  - 净值型组合创建、编辑、详情
- `ProductionAccountService`
  - 删除时真正清理产品
- `FundAssetComponent`
  - 投资范围内基金列表
- `Mof2AmsDataService`
  - `mofData` 策略指数与相关数据
- `RebalanceRecordService`
  - 再平衡记录分页查询

### 6.2 数据边界结论

这一层已经不是纯前端页面链路，而是显式进入：

- 业务账户层
- 组合主数据层
- 投后产品层

## 7. 公式与口径

这篇文档的高风险点不在数学公式，而在“对象语义”。

需要保留的业务口径包括：

- `accountId` 与 `portfolioId` 是并存语义，不要简单合并
- `momId` 是创建 / 编辑前提
- `reCal` 是显式业务决定
- 落地组合后要继承投后产品删除与权限体系

## 8. 重构时必须保留的契约

- `PmsDataAccountMapping` 的双标识关系必须保留。
- 创建、编辑、删除都要接入真实组合主数据体系，不能做成“研究库副本”。
- `editQuery -> edit/{reCal}` 的两阶段编辑协议不能省。
- `assetallocation` 和 `netValue` 两条链不能在第一步重构里粗暴合并。
