# 私募基金市场专题

## 1. 这篇文档解决什么问题

私募市场页和公募市场页表面相似，都是“左侧导航 + 右侧卡片”的专题页，但它的数据域和接口命名完全不同。

这篇文档重点梳理：

- `/app/fund/market/private` 的页面结构
- 卡片 inventory
- 每张卡片对应的 `rof_api` wrapper
- 当前仓库能追到的实现边界

## 2. 页面入口与整体结构

### 2.1 路由入口

关键路由：

- `/app/fund/market/private`

关键文件：

- `mof-web-fe/js/pages/fund-market/router.ts`
- `mof-web-fe/js/pages/fund-market/private-market/index.tsx`

页面结构和公募市场类似：

1. `PublicDataWrapper`
2. `LeftNav`
3. `RightContent`

页面初始化时同样会先加载基金公司公共数据：

- `GET ${rof_api}/api/fund/market/company/all`

### 2.2 菜单结构

菜单定义在：

- `mof-web-fe/js/pages/fund-market/private-market/template.ts`

当前确认三个一级菜单组：

- 收益统计
- 发行统计
- 私募报告

其中收益统计和私募报告都受 `auth.checkPrivateAuth()` 控制。

当前确认的 card inventory：

- `IssuanceFundType`
- `IssuingByInvestmentStrategy`
- `IssuingByManage`
- `IssuingByStructure`
- `RevenueFundType`
- `ReturnByInvestmentStrategy`
- `ReturnStructure`
- `WeekPrivateReport`

重构时要注意：

- 私募市场页不是“公募市场页换一组字段”
- 而是完全另一组 `rof` 接口命名体系

## 4. 后端实现边界

### 4.1 市场统计主域仍然是 `rof_api`

从前端常量与 wrapper 可明确确认：

- `private-market/constant.ts` 中 `apiHost = _config.rof_api`
- 私募市场统计使用的是 `/api/private/market/*`

与公募市场的 `/api/fund/market/*` 相比，私募市场已经切成另一组 API 命名空间。

### 4.2 当前仓库没有直接定位到这些 controller

截至当前工作区源码：

- 能清楚定位前端 wrapper
- 能清楚定位精确的 `rof` 端点
- 但在 `mom`、`mom-robo`、`brain`、`pms` 本地源码中没有直接看到这些 `private market` controller

因此当前阶段最稳妥的知识结论是：

- 私募市场统计主逻辑在 `rof` 域
- 目前本仓更适合保留“页面入口 + wrapper + 参数协议 + card 结构”知识
- 不适合臆测其内部是 `mom` service 或 `brain` 任务

## 5. 公式与数据口径

私募市场页的核心口径不是复杂数学公式，而是统计维度和时间点定义：

- `startPoint`
- `statisticsFrequency`
- `statisticsPoint`

其中尤其要注意：

- 收益统计类卡片通常同时区分“统计数值”和“分位值”
- 发行统计类卡片更偏数量、结构、管理方式分布
- 这两类卡片虽然 UI 接近，但接口域已经完全拆开

## 6. 重构时必须保留的契约

- 保留“发行统计”和“收益统计”两套接口族，不要硬并成一个通用 market endpoint。
- 保留私募周报 `WeekPrivateReport` 作为单独卡片，不要和统计表图混成一个模块。
- 保留 `startPoint / statisticsFrequency / statisticsPoint` 这类时间口径参数，它们是前后端共同的稳定协议。
- 对 `rof_api` 调用建议单独抽成 adapter，和 `mom`、`pmsLite` 内部接口分层管理。

## 7. 后续补充建议

- 继续补 `WeekPrivateReport` 的真实数据来源。
- 如果后续拿到 `rof` 私募市场服务仓，再补 controller / service / 数据源。
- 补一张“私募市场卡片 -> 时间参数 -> 图表维度”速查表，方便未来 Java / Rust / Go 重构时复用。
