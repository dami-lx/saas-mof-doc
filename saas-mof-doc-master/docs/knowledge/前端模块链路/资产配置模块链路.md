# 资产配置模块链路

## 1. 模块定位

这里的“资产配置”模块主要指前端中以资产配置模型、方案构建、组合创建和编辑为核心的页面，不包含情景分析和智能 FOF 推荐页。

当前更适合归到这一模块的入口有：

- `/app/asset/create`
- `/app/asset/edit/:id`
- `/app/topbottom/*`
- `/app/fund-allocation/*`

从职责上看，这个模块不是纯展示页，而是“创建组合方案并落地成产品/组合”的业务层。

## 2. 前端页面入口

路由定义：

- `mof-web-fe/js/pages/asset_new/router.ts`

已确认路由：

- `/app/asset/create`
- `/app/asset/edit/:id`
- `/app/topbottom/create`
- `/app/topbottom/edit/:id`
- `/app/topbottom/detail/:id`
- `/app/topbottom/portfolio-model/save/:id`
- `/app/fund-allocation/create`
- `/app/fund-allocation/edit/:id`
- `/app/fund-allocation/detail/:id`
- `/app/fund-allocation/portfolio-model/save/:id`

这些路由说明“资产配置模块”内部又分成几层：

- 研究 / 创建页
- 详情 / 结果页
- 保存为组合 / 产品页

## 3. 前端 API 入口

当前从源码中已确认的接口包括：

- `POST ${mof_api}/api/assetAllocation/models`
  - 资产配置研究数据
- `DELETE ${mof_api}/api/assetAllocation/{id}`
- `POST ${mof_api}/api/assetAllocation/calculate`
  - 在情景推荐链路里也被复用
- `GET ${mof_api}/api/assetAllocation/options`
  - 配置和可选项
- `POST ${mof_api}/api/assetAllocation/calcRebalancePeriods/v2`
  - 再平衡周期计算

从前端常量映射还能确认：

- `TRIGGER_SOURCE.bottom_up_asset_allocation`
- `TRIGGER_SOURCE.up_bottom_asset_allocation`

最终都会映射到：

- `/api/assetallocation`

对应文件：

- `mof-web-fe/js/pages/asset_new/*`
- `mof-web-fe/js/pages/topbottom/*`
- `mof-web-fe/js/v2/pages/Portfolio/overview/constants.ts`
- `mof-web-fe/js/v2/pages/Portfolio/overview/portfolio/editPortfolio/service.ts`

## 5. 实现语义

从控制器实现可以确认，这一模块的核心不是简单页面配置，而是：

1. 前端填写方案参数
2. `mom` 在创建时生成 `mom account`
3. 同时建立 `pms portfolio` 映射
4. 后续编辑时根据是否影响组合类型 / benchmark 决定是否重新计算
5. 再平衡、子类资产分配等能力会继续走算法链

其中一个非常关键的数据结构是：

- `PmsDataAccountMapping`

这说明资产配置页最终落地的是“业务账户 + PMS 组合”的双标识体系。

## 6. 下游依赖与数据来源

### 6.1 直接依赖

- `ProductionAccountService`
  - 删除组合时被复用，说明资产配置产物最终还是纳入投后产品体系
- `SubAssetAllocationComputer`
  - 子类资产算法
- `Mof2AmsDataService`
  - 获取 `mof` 策略指数 / 配置信息

### 6.2 间接依赖

- `pmsLite`
  - 创建和维护实际组合主数据
- `brain`
  - 对回测、再平衡、风险收益类指标提供算法支持

## 7. 公式与计算口径

这一模块的公式入口当前主要集中在两层：

- `mom` 内：
  - `SubAssetAllocationComputer`
  - 资产配置参数与再平衡规则
- `brain` 内：
  - 资产配置 / efficient frontier / rebalance 类端点

当前已确认的关键认知：

- 前端的“资产配置”不是公式所在处，前端只是参数装配层。
- 公式真正应该在 `mom` 的组合创建 / 再平衡服务和 `brain` 的算法端点里继续追踪。

## 8. 重构时必须保留的契约

- 创建接口返回的 `PmsDataAccountMapping` 语义不能丢。
- 编辑时 `reCal` 的判定规则不能只看前端字段变化，而要保留组合类别和 benchmark 变更语义。
- 资产配置方案最终会进入组合 / 产品体系，因此不能孤立成前端纯草稿对象。
