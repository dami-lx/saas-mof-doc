## 1. 模块定位

“私募基金模块”与公募基金模块结构类似，也包含：

- 私募基金筛选页
- 私募市场页
- 私募基金报告页

区别在于：

- 私募字段和结果后处理逻辑与公募不同
- 私募筛选在结果加工阶段有更强的业务口径差异
- 私募报告虽然进入同一报告平台，但卡片口径往往不同于公募

## 2. 前端页面入口

关键路由来自：

- `mof-web-fe/js/pages/product/router.ts`
- `mof-web-fe/js/pages/fund-market/router.ts`
- `mof-web-fe/js/v2/routes.ts`

当前确认的主要入口：

- `/app/fund/product/filter/private`
  - 页面目录：`mof-web-fe/js/pages/product/filter/privateV2`
- `/app/fund/market/private`
  - 页面目录：`mof-web-fe/js/pages/fund-market/private-market`
- `/report/:id(PRIVATE-.*)`
  - 页面容器：`mof-web-fe/js/v2/pages/IndependentReport/PageRouter`

## 2.1 当前已拆出的子域

为了便于重构，私募基金模块当前同样拆成三块看：

- 筛选平台
- 市场专题平台
- 报告 / 卡片平台

其中市场专题能力见：

- [私募基金市场专题](./私募基金-市场专题.md)

## 3. 前端 API 入口

关键文件：

- `mof-web-fe/js/pages/product/filter/privateV2/service.ts`
- `mof-web-fe/js/pages/product/filter/privateV2/components/AdvanceSearchPage/service.ts`

已确认的接口：

- `GET ${rof_api}/api/account/pfund/filter/field/dynamic`
- `GET ${rof_api}/api/account/pfund/plans`
- `POST ${rof_api}/api/account/pfund/plans`
- `PUT ${rof_api}/api/account/pfund/plans/{id}`
- `DELETE ${rof_api}/api/account/pfund/plans/{id}`
- `POST ${rof_api}/api/account/pfund/associateSearch`
- `POST ${rof_api}/api/account/pfund/advanceSearch`

补充接口：

- 持仓模糊搜索：
  - `/api/account/pfund/fuzzy/stock`
  - `/api/account/pfund/fuzzy/fund`

私募市场页则不走这套筛选接口，而是主要走 `rof_api` 的 `/api/private/market/*` 统计接口。

## 4. `mom` 后端入口

主 controller：

- `mom/mom-web/src/main/java/com/datayes/web/mom/account/privatefund/controller/PrivateFundSearchController.java`

已确认接口：

- `POST /api/account/pfund/ordinarySearch`
- `POST /api/account/pfund/advanceSearch`
- `POST /api/account/pfund/dynamicSearch`
- `POST /api/account/pfund/relatedSearch`
- `POST /api/account/pfund/searchByKey`

实现链路上的关键步骤：

1. `PrivateFundReqPreProcessor`
2. `PrivateFundAdvanceReqPreProcessor`
3. `PrivateAccountRankServiceImpl` / `PrivateAccountAdvanceRankServiceImpl`
4. `PrivateAccountListServiceImpl` / `PrivateAccountAdvanceListServiceImpl`
5. `PrivateResPostProcessor` / `PrivateDynamicResPostProcessor`

与公募基金的差异点：

- 私募筛选有专用的动态结果后处理器 `PrivateDynamicResPostProcessor`
- 还提供了按 symbol/name/recordCd 的 relatedSearch 和按关键词 searchByKey

## 5. 下游依赖与数据来源

### 5.1 私募筛选链路

当前可确认：

- 前端主要走 `rof_api`
- 后端在 `mom` 中完成预处理、排名、列表查询和结果格式化
- `AccountCache` 已参与相关搜索与精确 / 模糊匹配

### 5.2 私募市场链路

当前可确认：

- 页面级公共数据仍走 `GET ${rof_api}/api/fund/market/company/all`
- 发行统计走 `/api/private/market/stat/*`
- 收益统计走 `/api/private/market/profit/*`
- 当前仓未直接定位到这些 controller 的服务端实现

### 5.3 私募报告链路

私募报告页与公募报告页共享统一报告平台：

- 前端报告容器：`js/v2/pages/IndependentReport/PageRouter`
- 后端卡片入口：`/api/card`

但卡片口径与模板可能不同，因此后续应该补：

- `PRIVATE-*` 路由对应的模板选择逻辑
- 私募专属卡片与 `brain` 端点的映射

## 6. 公式与计算口径

- 私募筛选页主要还是查询 / 排名型链路，重点是字段与排序口径。
- 私募报告页的公式继续下沉到卡片模板与 `brain`，不应在“私募筛选模块”里混写。

## 7. 重构时必须保留的契约

- 私募筛选与公募筛选虽然结构相似，但不能简单共用同一请求模型。
- 私募市场页和公募市场页也不能简单复用同一个 market service。
- `PRIVATE-*` 报告和 `MUTUAL-*` 报告应保持模板级分流能力。
- `relatedSearch`、`searchByKey` 这类补充接口容易在重构时被忽略，但它们直接影响前端搜索体验。

## 8. 待继续补充

- 私募市场页的专题 API 清单。
- 私募报告常用卡片与 `brain` 计算端点映射。
- 私募筛选字段与缓存 / 搜索索引字段对照表。

## 9. 当前拆解建议

私募基金模块后续同样建议拆成：

1. 筛选平台
2. 市场专题平台
3. 报告平台

其中尤其要注意：

- 私募筛选字段体系与公募不同
- 报告容器可能共用，但模板和卡片口径不能混并
