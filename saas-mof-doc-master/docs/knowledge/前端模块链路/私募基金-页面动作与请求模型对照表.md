# 私募基金页面动作与请求模型对照表

## 1. 这篇文档解决什么问题

私募基金筛选和公募看上去页面类似，但字段、动态字段来源和搜索接口并不完全相同。

这篇文档把私募筛选页的常见动作直接映射到后端请求模型和实现类，方便做同构重写。

## 2. 页面动作对照

## 3. 关键请求模型速记

### 3.1 `PrivateFundListQueryRequest`

这是私募普通筛选主请求模型，继承 `PageBaseRequest`，默认内置：

- `externalSource = PRIVATE`

常见字段可分为几类：

- 基础信息：
  - `childStrategy`
  - `openDate`
  - `netValueDate`
  - `navFrequency`
  - `recordDate`
  - `recordStatus`
  - `runningStatus`
- 绩效指标：
  - `annualTotalReturn`
  - `annualTotalRisk`
  - `maxDrawdown`
  - `sharpRatio`
  - `netValue`
- 管理人与机构特征：
  - `managePublicSentiment`
  - `consultManageSize`
  - `instOpenDate`
  - `managerBackground`
- 产品结构与重仓信息：
  - `heavyHoldingStockCode`
  - `heavyHoldingFundCode`
  - `structured`
  - `macFund`
- 搜索与扩展：
  - `funnelField`
  - `searchbarField`
  - `dynamicSearchField`
  - `customStrategy`

### 3.2 `AssociateSearchRequest`

私募联想搜索与公募共用同一请求模型：

- `associateSearchSource`
- `associateSearchField`

前端典型用法仍然是传：

- `name`
- `op = LIKE`
- `values = [keyword]`

### 3.3 `QueryPlanCreateRequest`

私募方案保存与公募同构，核心字段也是：

- `name`
- `desc`
- `planContent`
- `template`

差异主要在保存时映射到的 `UserConfigType` 不同：

- `USER_PFUND_QUERY_PLAN`
- `USER_PFUND_QUERY_SHOW_TEMPLATE`

## 4. 关键实现结论

- 私募筛选主入口集中在：
  - `PrivateFundSearchController`
  - `PrivateFundDynamicFilterController`
  - `PrivateFundAssociateSearchController`
  - `PrivateFundQueryPlanController`
- 普通筛选、高级筛选、动态搜索同样是拆开的，不是单一搜索接口。
- 私募动态筛选字段同样有缓存语义，注释明确写明会接入 `DynamicFilterFieldCache` 的统一刷新流程。
- 私募筛选相比公募，多了：
  - `relatedSearch`
  - `searchByKey`
  这类更偏账户缓存检索的辅助接口。
- 方案保存虽然结构相似，但 `UserConfigType` 和公募不同，重构时不要混用。

## 5. 重构时优先保留什么

- 保留公募 / 私募请求模型分离，不要试图在第一步重构时合成一个统一 FundSearchRequest。
- 动态字段接口继续单独保留，并维护缓存刷新逻辑。
- `relatedSearch` 与 `searchByKey` 这类私募特有辅助接口要单独收口，不要误删。
- 方案存储要继续区分公募和私募的 `UserConfigType`。
