# 资产配置研究与创建专题

## 1. 这个专题关注什么

这篇文档只看资产配置里的“研究型链路”，不讨论最终落地成产品后的编辑和投后管理。

它重点回答：

- 前端从哪些页面进入资产配置研究与方案创建
- 研究页、模型页、试算页分别调用哪些接口
- `mom` 里谁负责模型管理，谁负责真正发起算法计算
- 数据和参数是落在哪些存储或算法进程上

## 2. 前端页面入口

关键路由：

- `/app/asset/create`
- `/app/topbottom/create`
- `/app/fund-allocation/create`

主要目录：

- `mof-web-fe/js/pages/asset_new`
- `mof-web-fe/js/pages/topbottom`
- `mof-web-fe/js/pages/fund-allocation`

从页面结构上，这条链路通常会经过三层：

- 研究列表 / 模型列表
- 参数编辑与方案试算
- 将研究结果保存为“模型”

## 3. 前端 API 入口

### 3.1 研究列表与模型管理

关键文件：

- `mof-web-fe/js/pages/asset_new/research/actions.ts`
- `mof-web-fe/js/pages/topbottom/services.ts`
- `mof-web-fe/js/pages/topbottom/components/AssetModel/modelToPlan/actions.ts`

已确认接口：

- `POST ${mof_api}/api/assetAllocation/models`
- `GET ${mof_api}/api/assetAllocation/model/{id}`
- `POST ${mof_api}/api/assetAllocation/save`
- `DELETE ${mof_api}/api/assetAllocation/{id}`

### 3.2 参数选项、再平衡日期与持仓试算

关键文件：

- `mof-web-fe/js/pages/fund-allocation/create/services.ts`
- `mof-web-fe/js/pages/topbottom/components/AssetModel/modelToPlan/actions.ts`

已确认接口：

- `GET ${mof_api}/api/assetAllocation/options`
- `POST ${mof_api}/api/assetAllocation/calcRebalancePeriods/v3`
- `POST ${mof_api}/api/assetAllocation/calculate`
- `POST ${mof_api}/api/assetAllocation/eFrontier`

前端的重要实现事实：

- `format2BeModelParams` 会把前端模型参数转换成后端 `modelType + xxxParams` 结构。
- `format2BeAssets` 会统一把资产边界、风险权重、固定收益产品收益率转成后端小数口径。
- `getPollingResult` 会轮询 `GET ${mof_api}/api/task/{taskId}`，直到异步计算完成。

### 3.3 研究页的“公式”不在前端

前端虽然有不少参数转换逻辑，但它们本质上只是“参数装配器”：

- 风险平价、均值方差、Black-Litterman 的公式并不在前端执行
- 前端负责把周期、自定义区间、benchmark、风险偏好等转换成后端约定 JSON
- 真正的计算入口在 `mom` 里的 process / uranus 链路

## 4. `mom` 后端入口

主 controller：

- `mom/mom-web/src/main/java/com/datayes/web/mom/controller/AssetAllocationController.java`

### 4.1 模型与配置类接口

已确认接口：

- `GET /api/assetAllocation/options`
- `GET /api/assetAllocation/model/{id}`
- `POST /api/assetAllocation/models`
- `POST /api/assetAllocation/save`
- `DELETE /api/assetAllocation/{id}`

实现语义：

- `options` 直接来自模板文件 `json/asset_allocation.json`
- `model/{id}` 先查 `AssetAllocationMapper`，再通过 `BigDataParamsService` 回填大参数
- `save` 会把模型元数据和大参数拆开存：
  - 元数据落 `AssetAllocationMapper`
  - 大参数落 `BigDataParamsService`

这意味着：

- 资产配置模型不是单表结构
- “模型壳子”和“参数正文”是分开持久化的

### 4.2 研究计算与有效前沿

已确认接口：

- `POST /api/assetAllocation/calcRebalancePeriods/v2`
- `POST /api/assetAllocation/calcRebalancePeriods/v3`
- `POST /api/assetAllocation/calculate`
- `POST /api/assetAllocation/eFrontier`

实现语义：

- `calcRebalancePeriods/v2` 和 `v3` 都委托 `SmartFofParamModificationService`
- `calculate` 会把 `mustUseMajor` 注入请求，再调用 `AssetAllocationProcess.request(request, userId)`
- `eFrontier` 会补 tenant，并为 `ACCOUNT` 类资产补 `dbType="mom"`，再调用 `AssetEfficientFrontierProcess.request`

关键结论：

- `calculate` / `eFrontier` 是异步任务入口，不是同步算法函数
- 研究页的大类资产计算，已经进入 `process -> task -> polling` 模式

## 5. 数据来源与外部依赖

### 5.1 当前可确认依赖

- `ITradeDateCache`
  - 再平衡日期回推到交易日
- `AssetAllocationMapper`
  - 模型元数据查询与保存
- `BigDataParamsService`
  - 模型大参数持久化
- `AssetAllocationProcess`
  - 大类资产试算任务入口
- `AssetEfficientFrontierProcess`
  - 有效前沿任务入口

### 5.2 外部能力边界

从 controller 职责看，`mom` 在这里扮演的是“研究平台编排层”：

- 模型模板和参数由 `mom` 管
- 交易日、选项和用户模型由 `mom` 管
- 真正的资产配置解算通过 process 链路继续下钻到算法服务

## 6. 公式与口径

这一专题最重要的口径不是最终算法公式，而是“研究参数协议”。

当前已确认的稳定契约：

- `modelType` 决定后端读取哪组 `xxxParams`
- 风险收益率、边界、权重一律转成小数口径再下发
- 自定义协方差区间 / 收益区间要拆成：
  - `isCustomized...`
  - `BeginDate`
  - `EndDate`
- `eFrontier` 和普通 `calculate` 的请求结构相似，但职责不同

## 7. 重构时必须保留的契约

- 不要把“模型存储”和“算法计算”合并成一个服务函数，它们现在是明显两层。
- 不能丢掉 `BigDataParamsService` 这种“大参数外置存储”语义，否则历史模型兼容会出问题。
- `task + polling` 协议要保留，因为前端明确依赖它。
- `assetAllocation` 与 `assetallocation` 两套路径不是大小写噪音，而是研究链和落地链的边界。
