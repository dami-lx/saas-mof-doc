# 资产配置子类资产与再平衡专题

## 1. 这个专题关注什么

这篇文档只看资产配置中的“二次计算链路”：

- 子类资产分配
- 再平衡日期推导
- 再平衡记录与批量触发

它是资产配置里最接近算法执行层的一段，但前端仍然通过异步任务方式消费。

## 2. 前端页面入口

主要目录：

- `mof-web-fe/js/pages/topbottom/components/ChildAssetModel`
- `mof-web-fe/js/pages/fund-allocation/create`
- `mof-web-fe/js/pages/topbottom/components/AssetModel/modelToPlan`

对应页面能力：

- 大类资产方案确定后，继续求子类资产持仓
- 根据起始日 / 再平衡规则生成日期列表
- 查看历史再平衡记录

## 3. 前端 API 入口

### 3.1 子类资产分配

关键文件：

- `mof-web-fe/js/pages/topbottom/components/ChildAssetModel/services.ts`

已确认接口：

- `POST ${mof_api}/api/assetallocation/subAssetAllocation/v2`
- `GET ${mof_api}/api/task/{taskId}`

实现语义：

- 前端先提交 `SubParamPost`
- 后端立即返回 `taskID`
- 前端轮询任务，最终只消费 `result.holdings`

### 3.2 再平衡日期

关键文件：

- `mof-web-fe/js/pages/fund-allocation/create/services.ts`
- `mof-web-fe/js/pages/topbottom/components/AssetModel/modelToPlan/actions.ts`
- `mof-web-fe/js/pages/aiFof/IntelligentFof/components/Params/services.ts`

已确认接口：

- `POST ${mof_api}/api/assetAllocation/calcRebalancePeriods/v2`
- `POST ${mof_api}/api/assetAllocation/calcRebalancePeriods/v3`

当前已确认口径：

- `v2` 更接近按日期 / 月份规则推导
- `v3` 明确支持基于“天数”的再平衡周期

### 3.3 再平衡记录与运维触发

后端已确认接口：

- `GET /api/assetallocation/rebalanceRecord/{id}`
- `POST /api/assetAllocation/balance`
- `POST /api/assetAllocation/smartFof/balance`

前端当前已明确消费的是：

- `rebalanceRecord`

后两者更像运维 / 批处理入口。

## 4. `mom` 后端入口

### 4.1 子类资产算法入口

主 controller：

- `mom/mom-web/src/main/java/com/datayes/web/mom/assetallocation/upbottom/controller/UpBottomAssetAllocationController.java`
- `mom/mom-web/src/main/java/com/datayes/web/mom/assetallocation/smartfof/controller/SmartFofManagementController.java`

核心实现：

- `SubAssetAllocationComputer.requestV2(request, userId, tenant)`

关键结论：

- 资产配置和智能 FOF 复用了同一套子类资产算法入口
- 子类资产分配本质上是共享算法平台能力，不是单业务私有逻辑

### 4.2 再平衡日期入口

主 controller：

- `mom/mom-web/src/main/java/com/datayes/web/mom/controller/AssetAllocationController.java`

核心实现：

- `calcRebalancePeriods/v2 -> SmartFofParamModificationService.calculateRebalancePeriods`
- `calcRebalancePeriods/v3 -> SmartFofParamModificationService.calculateRebalancePeriodsOfDay`

这说明：

- 再平衡日期计算并不只服务智能 FOF
- 资产配置和智能 FOF 在这一层已经共享实现

## 5. 数据来源与外部依赖

### 5.1 当前可确认依赖

- `SubAssetAllocationComputer`
  - 子类资产算法任务入口
- `ITradeDateCache`
  - 交易日回推
- `SmartFofParamModificationService`
  - 再平衡周期计算
- `RebalanceRecordService`
  - 再平衡记录分页查询

### 5.2 任务模式

这里的所有关键接口都在强化一个事实：

- 计算不是同步返回
- 页面依赖 `task / polling` 协议
- 前端拿到的是“最终 holdings”而不是中间解题过程

## 6. 公式与口径

这一专题最应该保留的是“时间和持仓口径”：

- 再平衡日期必须回到交易日
- 子类资产结果最终对齐到 `holdings`
- 再平衡规则至少区分：
  - 指定日期
  - 周期规则
  - 动态再平衡

另外，前端 `format2BeRebalanceParams` 已经明确约定了字段：

- `isDynamic`
- `rebalanceRule`
- `rebalanceDateList`
- `begin`

这套请求结构未来最好保留为兼容层。

## 7. 重构时必须保留的契约

- 子类资产分配不要嵌回前端或同步接口，继续保留异步计算语义更稳妥。
- 再平衡日期的交易日修正规则不能丢。
- `v2` / `v3` 两套再平衡日期接口要先搞清楚调用方，再决定是否合并。
- 子类资产结果必须继续对齐 `holdings` 协议，否则后续页面展示会断。
