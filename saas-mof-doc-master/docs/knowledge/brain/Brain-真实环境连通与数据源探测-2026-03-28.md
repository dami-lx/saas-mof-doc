# Brain 真实环境连通与数据源探测（2026-03-28）

## 1. 这篇文档的用途

这篇文档记录当前 `brain` 重构过程中已经验证过的真实环境信息，目标不是保存密码，而是固定下面这些“对未来调试很重要”的事实：

- 哪些 MySQL / Redis 环境当前可达
- 哪些环境是 `brain` 股票归因主链真正依赖的
- 哪些环境虽然出现在配置里，但不是当前运行时主路径的硬依赖
- 当前 Redis 里的真实 key 命名与 cluster 行为

## 2. 当前已验证的可用环境

### 2.0 Brain QA 环境

已确认：

- 域名: `brain-web.respool2.wmcloud-qa.com`
- 端口: 80
- 无需认证
- API 前缀: `/api/`
- 任务提交: POST `/api/common/holding_return_contribution` 等接口
- 任务查询: GET `/api/task/{task_id}`

这意味着：

- 可以通过域名直接调用 brain API 获取预期输出
- 接口返回 task_id，需要轮询任务状态获取结果

### 2.1 Datayes MySQL

已确认：

- 原域名 `db-dydbdev-ro.wmcloud.com`
- 当前可解析 IP 为 `10.22.220.50`
- `10.22.220.50:3313` 可成功连接并执行查询

已验证的数据对象包括：

- `md_trade_cal`
- `md_security`
- `IDX`
- `IDX_MAPPING_MOM`

这意味着：

- 股票归因链路里涉及的交易日、证券映射、benchmark 映射等基础数据可以直接从这个环境读取
- 后续做 snapshot exporter 时，Datayes MySQL 可以作为稳定数据源之一

### 2.2 Redis Cluster

已确认：

- `10.24.95.67:6379`
- `10.24.95.68:6379`
- `10.24.95.69:6379`

这三个节点都能：

- TCP 连通
- AUTH 成功
- PING 成功

同时已经确认它是 **Redis Cluster**，不是单机 Redis。

因此：

- 后续所有 exporter / probe / debug 脚本都必须具备 cluster-aware 行为
- 不能假设单节点 `GET` 就能拿到完整数据

## 3. 当前不稳定或不可直接依赖的环境

### 3.1 MySQL riskmdl

已确认：

- `10.27.30.247:3310` TCP 可以建立连接
- 但执行简单查询时会报 `Lost connection to MySQL server during query`

这说明：

- 它不是“完全网络不通”
- 但也不是当前可以放心作为运行期依赖的稳定入口

在重构设计上应当把它视为：

- 可能的上游原始数据源
- 不是当前股票 Brinson 主链运行时的首选依赖

## 4. 股票归因主链当前真正依赖什么

从 `brain` 现有代码和本次探测结果综合看，股票归因主链的运行时依赖更接近：

1. Datayes MySQL
   - 负责交易日、证券主键映射、benchmark 映射等基础数据
2. Redis Cluster
   - 负责股票收益、行业、benchmark 成分、风险模型缓存

而不是：

- 运行时每次都直接打 `riskmdl` MySQL

## 5. 请求主键与外部数据主键的关系

已对 `brain-brinson-test/a-brinson-request.json` 中的样例确认：

- `holding.consPositionId`
- 可以直接映射到 `md_security.SECURITY_ID`

这意味着：

- 请求层的证券主键可以直接作为 Redis / 风险模型数据 join key
- 重构后的 `SecurityUniverse` 不需要额外引入第二套证券主键体系

## 6. benchmark id 的真实语义

这是一个非常容易误解的点。

对于请求中的 benchmark `1782`：

- 业务传入的 `1782`
- 在 `brain` 的旧 Python 代码里，是按 `IDX_MAPPING_MOM.SECURITY_ID` 使用的
- 不是按 `IDX_MAPPING_MOM.ID` 使用的

已验证结果：

- `SECURITY_ID = 1782`
- `SEC_SHORT_NAME = 沪深300`
- `WEIGHT_LOCATED_TABLE = csi_idx_weight`
- 对应的映射表主键其实是 `ID = 158`

这对未来重构非常关键，因为如果把 benchmark id 错当成映射表主键，后面的权重表定位会整体跑偏。

## 7. Redis 中已确认存在的关键数据

本次已经确认存在真实 payload 的 key 包括：

- `stock:ret:d:{td}`
- `ind:SW1:21:{td}`
- `bm:1782:{td}`
- `fret_sw21:d:{td}`
- `sret_sw21:d:{td}`
- `rm_sw21:exp:{td}`
- `rm_sw21:cov:short:{td}`
- `rm_sw21:srisk:short:{td}`

这里可以看出两个事实：

### 7.1 业务行业口径与 Redis key 口径不是同一字符串

- 业务里常见：`SW21_1`
- Redis 里实际 key：`ind:SW1:21:{td}`

所以未来 provider 层必须保留结构化字段：

- category = `SW`
- version = `21`
- level = `1`

### 7.2 风险模型运行时数据确实在 Redis 中

这再次说明股票归因运行时更偏缓存驱动，而不是 riskmdl 数据库直查驱动。

## 8. 当前真正的兼容阻碍

现在阻碍我们直接做在线读取的，不是“数据不存在”，而是旧序列化格式兼容问题。

已确认的典型错误包括：

- `No module named 'pandas.core.indexes.numeric'`
- `Argument 'placement' has incorrect type (expected pandas._libs.internals.BlockPlacement, got slice)`

这意味着：

- Redis 中的对象由较老 pandas 版本序列化
- 当前本地环境里的 pandas 版本已经和历史 pickle 结构不兼容

所以后续应优先采用：

- 旧环境导出 snapshot
- 新环境消费 snapshot

而不是要求 Rust 侧直接兼容历史 pandas pickle。

## 9. 结论

当前对 `brain` 股票归因主链来说，短期内可以视为稳定可用的环境是：

- Datayes MySQL
- Redis Cluster

`riskmdl` MySQL 虽然在配置中存在，也在底层 loader 中被引用，但目前不应被视为当前运行主链的稳定依赖入口。
